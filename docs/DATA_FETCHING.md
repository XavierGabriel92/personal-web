## Data Fetching Strategy

### ✅ Always use Suspense queries + `<Suspense>` boundaries

All data fetches in this app use the `*Suspense` variant of the generated hooks (e.g. `useGetApiClientsSuspense`, `useGetApiRoutineByIdSuspense`). These hooks throw a Promise when data is not yet available, which React catches at the nearest `<Suspense>` boundary to show a fallback.

**Every component that calls a `*Suspense` hook must be wrapped in a `<Suspense>` boundary**, either directly or by a parent. Omitting the boundary causes a blank-page blink during navigation while TanStack Router's internal Suspense catches the throw with no visible fallback.

```tsx
// ✅ Correct
import { Suspense } from 'react';
import { Spinner } from '@/components/ui/spinner';
import { useGetApiClientsSuspense } from '@/gen';

function ClientsList() {
  const { data } = useGetApiClientsSuspense(); // throws until data is ready
  return <div>{/* render */}</div>;
}

function RouteComponent() {
  return (
    <Suspense fallback={<Spinner className="size-8" />}>
      <ClientsList />
    </Suspense>
  );
}

// ❌ Wrong — causes blank-page blink
function RouteComponent() {
  return <ClientsList />; // no Suspense boundary
}
```

### Where to place the `<Suspense>` boundary

Place the boundary at the **route component level** (in the `src/routes/` file), not deep inside page or component files. This keeps page/component files clean and makes the loading strategy visible at the routing layer.

```tsx
// src/routes/trainer/clients/$clientId/route.tsx
function RouteComponent() {
  const { clientId } = Route.useParams();
  return (
    <>
      <Suspense fallback={<Spinner className="size-8" />}>
        <TrainerLayoutClient clientId={clientId} />
      </Suspense>
      <Outlet />
    </>
  );
}
```

When a route renders multiple independent suspending sections, wrap each one separately so they don't block each other.

### Use the `<Spinner>` component as the fallback

Always use `@/components/ui/spinner` as the `<Suspense>` fallback for consistency:

```tsx
import { Spinner } from '@/components/ui/spinner';

<Suspense fallback={<Spinner className="size-8" />}>
  <MyComponent />
</Suspense>
```

### ❌ Avoid non-suspense query hooks for UI data

Do not use the non-suspense variants (`useGetApiRoutines`, `useGetApiClients`, etc.) for rendering UI data. They require manual `isLoading`/`error` checks in every component and produce inconsistent loading states.

```tsx
// ❌ Avoid for UI data
const { data, isLoading, error } = useGetApiRoutines();
if (isLoading) return <Spinner />;
if (error) return <ErrorMessage />;

// ✅ Prefer
const { data } = useGetApiRoutinesSuspense(); // clean, no branching
```

**Exception**: non-suspense hooks are acceptable when the query is conditional or optional (e.g. `enabled: !!id`) and the UI already handles the undefined state gracefully.

## Mutations

### Always reflect `isPending` in the UI

Every mutation must disable its trigger and show a visual loading indicator while `isPending` is true. Never let a user submit the same action twice.

```tsx
const { mutateAsync: createRoutine, isPending } = usePostApiRoutineCreate();

<Button onClick={handleSubmit} disabled={isPending}>
  {isPending ? <><Spinner /> Salvando...</> : "Salvar"}
</Button>
```

### Always handle `onSuccess` and `onError`

```tsx
const { mutateAsync: updateClient, isPending } = usePutApiClientById();

await updateClient({ id, data }, {
  onSuccess: async () => {
    toast.success("Cliente atualizado!");
    await queryClient.invalidateQueries({ queryKey: getApiClientsSuspenseQueryKey() });
  },
  onError: () => {
    toast.error("Erro ao atualizar. Tente novamente.");
  },
});
```

### Invalidate after mutations

After any create/update/delete, invalidate the affected query keys so the UI stays in sync:

```typescript
queryClient.invalidateQueries({ queryKey: getApiRoutinesQueryKey() });
```

For more precise updates (avoiding a full refetch), use `setQueryData` with the new value from the API response:

```typescript
queryClient.setQueryData(getApiRoutineByIdSuspenseQueryKey(routineId), updatedRoutine);
```

## Optimistic Updates

For drag-and-drop or other interactions that should feel instant, update the cache before the API call and rollback on error:

```typescript
const queryKey = getApiWorkoutByIdSuspenseQueryKey(workoutId);
const oldData = queryClient.getQueryData(queryKey);

// Optimistically update
queryClient.setQueryData(queryKey, (old) => ({ ...old, exercises }));

reorderExercises(
  { data: { exerciseWorkouts: exercisesData } },
  {
    onSuccess: (response) => {
      queryClient.setQueryData(queryKey, response);
      toast.success("Ordem atualizada!");
    },
    onError: () => {
      queryClient.setQueryData(queryKey, oldData); // rollback
      toast.error("Erro ao reordenar. Tente novamente.");
    },
  }
);
```

## Query Key Management

Query keys are generated by Kubb and exported alongside their hooks:

```typescript
import { getApiRoutinesQueryKey, getApiRoutineByIdSuspenseQueryKey } from "@/gen";

// Invalidate a list
queryClient.invalidateQueries({ queryKey: getApiRoutinesQueryKey() });

// Update a single item in cache
queryClient.setQueryData(getApiRoutineByIdSuspenseQueryKey(routineId), updatedRoutine);
```

## Session Management

The session uses a dedicated cached fetch (not a hook) for `beforeLoad` route guards:

```typescript
// src/hooks/auth.ts
export const cachedSession = () => {
  return queryClient.fetchQuery({
    queryKey: sessionQueryKey,
    queryFn: fetchSession,
    staleTime: 5 * 60 * 1000, // fresh for 5 minutes
  });
};
```

On first page load this hits the network; on subsequent navigations within 5 minutes it returns from cache synchronously.

## Best Practices

- ✅ Always use `*Suspense` hooks for UI data fetches
- ✅ Always wrap suspense-query components in `<Suspense fallback={<Spinner />}>`
- ✅ Place `<Suspense>` boundaries at the route component level (`src/routes/`)
- ✅ Always disable mutation triggers and show `<Spinner>` while `isPending`
- ✅ Always handle `onSuccess` (invalidate/update cache + toast) and `onError` (toast) on mutations
- ✅ Use optimistic updates for instant-feedback interactions (drag, toggles)
- ✅ Rollback optimistic updates on error
- ✅ Use generated query keys from Kubb — never hardcode them
- ✅ Use `staleTime` to avoid unnecessary refetches
